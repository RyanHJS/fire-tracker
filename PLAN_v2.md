# 🔥 FIRE Finance Tracker — Project Plan

The FIRE Finance Tracker is an interactive web application designed to help users demystify their path to early retirement. Its core functionality allows users to input their current financial metrics—such as age, portfolio balance, and annual retirement budget—and instantly calculates their target "FIRE Number" alongside an estimated retirement age using a dynamic, year-by-year simulation loop. The primary goal of the application is to provide an immediate, empowering "Aha!" moment, transforming abstract financial concepts into a concrete, personalized timeline that motivates users to optimize their savings and confidently plan for their financial independence.

---

## 1. Data Schema

The schema is split into two layers: **Core Entities** (user identity, accounts, transactions) and the **FIRE Plan Layer** (the goal configuration and retirement calculation logic). The FIRE Plan layer uses an inheritance model — a parent class holds fields shared by all plan types, and child classes extend it with type-specific variables.

---

### Core Entities

#### `users`

| Column | Type | Notes |
|---|---|---|
| `user_id` | `uuid` PK | Auto-generated by Supabase Auth |
| `name` | `text` | User's display name |
| `age` | `int` | Current age, entered during onboarding |
| `email` | `text` | Unique, used for login |
| `created_date` | `date` | Default: today |

---

#### `accounts`
One user can have multiple accounts.

| Column | Type | Notes |
|---|---|---|
| `account_id` | `uuid` PK | |
| `user_id` | `uuid` | FK → `users.user_id` |
| `account_name` | `text` | e.g. "Chase Checking", "Vanguard 401k" |
| `account_type` | `enum` | `Living` \| `Investment` |
| `current_balance` | `numeric(12,2)` | Automatically updated by a PostgreSQL trigger on every transaction insert |
| `is_active` | `bool` | Default: `true` |

> **Living** accounts represent day-to-day operational money (the equivalent of checking + savings combined). Only **Investment** accounts are summed into the portfolio balance used for FIRE calculations.
> `current_balance` is maintained by a **PostgreSQL trigger** (not application code) — whenever a row is inserted into `transactions`, the database instantly adds or subtracts the amount from the linked account. This prevents data mismatch and keeps the frontend code clean.

---

#### `transactions`
One account can have multiple transactions.

| Column | Type | Notes |
|---|---|---|
| `transaction_id` | `uuid` PK | |
| `account_id` | `uuid` | FK → `accounts.account_id` |
| `amount` | `numeric(12,2)` | Always positive — direction is set by `type` |
| `date` | `date` | When the transaction occurred |
| `type` | `enum` | `Income` \| `Expense` |

> The annual contribution used in FIRE forecasting is derived from transaction history: `SUM(Income) − SUM(Expense)` over the past 12 months across all active Investment accounts.

### FIRE Plan Layer

A user has a one-to-one relationship with a `FirePlan` by default. The design supports one-to-many in the future, allowing users to compare scenarios (e.g. "What if I do Barista FIRE vs. Traditional FIRE?").

All FIRE plan types share a common parent class with the base fields every plan needs. Child classes extend this with type-specific variables.

---

#### Parent Class: `FirePlan`

| Column | Type | Notes |
|---|---|---|
| `plan_id` | `uuid` PK | |
| `user_id` | `uuid` | FK → `users.user_id` |
| `plan_name` | `text` | e.g. "My Lean FIRE Goal" |
| `target_retirement_age` | `int` | The user's **goal** — the age they *want* to retire |
| `annual_retirement_budget` | `numeric(12,2)` | Planned annual spending in retirement |
| `safe_withdrawal_rate` | `numeric(5,4)` | Default `0.04` (4%), user-customizable |

> `current_age` is intentionally **not stored here**. It is always read from `users.age` (joined via `user_id`) to maintain a single source of truth and prevent the two tables from drifting out of sync.

> **Base FIRE Number** = `annual_retirement_budget / safe_withdrawal_rate`
> (Equivalent to `annual_retirement_budget × 25` at the default 4% SWR)

**`target_retirement_age` vs. `estimated_retirement_age`** — these are two distinct concepts:
- `target_retirement_age` (stored) — the age the user *wants* to retire. Set once during onboarding.
- `estimated_retirement_age` (computed, not stored) — the age they *will actually* retire based on current portfolio balance, contributions, and expected ROI. Recalculated live by `calculate_retirement_age()`.

The UI compares the two: if `estimated > target`, a warning is shown (e.g. *"You are off track by 3 years — consider increasing contributions"*). If `estimated ≤ target`, the user is on track.

Each plan type also inherits the `calculate_retirement_age()` method — documented in the [Calculation Logic](#calculation-logic) section below.

---

#### Child Class: `StandardFirePlan`
Handles **Traditional FIRE**, **Lean FIRE**, and **Fat FIRE**. All three use the same formula — the only difference is how large the user sets `annual_retirement_budget` (modest for Lean, generous for Fat).

*Inherits all fields from `FirePlan`, plus:*

| Column | Type | Notes |
|---|---|---|
| `plan_id` | `uuid` PK | FK → `fire_plans.plan_id` |
| `current_annual_income` | `numeric(12,2)` | Gross annual income — used as an onboarding estimate only |
| `current_savings_rate` | `numeric(5,4)` | Savings rate — used as an onboarding estimate only, e.g. `0.30` |

> **FIRE Number** = `annual_retirement_budget / safe_withdrawal_rate`

**Source of truth for contributions:** `current_annual_income` and `current_savings_rate` are collected at onboarding so the app can generate an immediate "what-if" forecast before the user has any transaction data. Once transactions exist, the annual contribution is derived directly from the `transactions` table (`SUM(Income) − SUM(Expense)` over 12 months) and overrides the static onboarding estimate. The transactions table is always the ultimate source of truth.

---

#### Child Class: `BaristaFirePlan`
The user plans to work part-time in semi-retirement. The portfolio only needs to cover the gap between expenses and part-time income.

*Inherits all fields from `FirePlan`, plus:*

| Column | Type | Notes |
|---|---|---|
| `plan_id` | `uuid` PK | FK → `fire_plans.plan_id` |
| `expected_part_time_income` | `numeric(12,2)` | Annual income from part-time work in semi-retirement |

> **FIRE Number** = `(annual_retirement_budget − expected_part_time_income) / safe_withdrawal_rate`

---

#### Child Class: `CoastFirePlan`
The user saves aggressively early until their portfolio is large enough to compound on its own to the full FIRE number by `target_retirement_age`, without any further contributions.

*Inherits all fields from `FirePlan`, plus:*

| Column | Type | Notes |
|---|---|---|
| `plan_id` | `uuid` PK | FK → `fire_plans.plan_id` |
| `assumed_annual_growth_rate` | `numeric(5,4)` | Expected annual market return, default `0.07` (7%) |

> **Full FIRE Target** = `annual_retirement_budget / safe_withdrawal_rate`
> **Coast FIRE Number** = `Full FIRE Target / (1 + assumed_annual_growth_rate) ^ (target_retirement_age − users.age)`
> The user has reached Coast FIRE when their current portfolio balance ≥ the Coast FIRE Number.
> `users.age` is joined from the `users` table — it is not stored on the plan itself.

### Calculation Logic

The core output of the app — the user's **estimated retirement age** — is computed by a `calculate_retirement_age()` method defined on the `FirePlan` parent class. Child classes can override it where the logic differs (e.g. `CoastFirePlan`).

#### Variables Required

| Variable | Source |
|---|---|
| **Target FIRE Number** | Computed from the active `FirePlan` child class formula |
| **Current Portfolio Balance** | SUM of `current_balance` across all active `Investment` accounts (maintained by PostgreSQL trigger) |
| **Annual Contribution** | Primary: `SUM(Income) − SUM(Expense)` over the past 12 months from `transactions`. Fallback: `current_annual_income × current_savings_rate` from `StandardFirePlan` if no transaction data exists yet. |
| **Expected ROI** | From plan config (`assumed_annual_growth_rate`), default `0.07` (7%) |

#### Simulation Loop Approach

Rather than a closed-form compound interest formula, `calculate_retirement_age()` uses a year-by-year simulation loop. This is easier to read, easier to debug, and allows for future enhancements (e.g. varying contribution amounts over time).

```
current_balance = total portfolio balance
year = current year

while current_balance < target_fire_number:
    current_balance = current_balance * (1 + expected_roi)
    current_balance = current_balance + annual_contribution
    year = year + 1

estimated_retirement_year = year
estimated_retirement_age  = users.age + (estimated_retirement_year - current_year)  # age joined from users table

# UI comparison:
if estimated_retirement_age > target_retirement_age:
    show warning: "Off track by N years — consider increasing contributions"
else:
    show confirmation: "On track to retire by your goal age"
```

> For `CoastFirePlan`, `annual_contribution` is set to `0` — the loop only applies compound growth, checking when the portfolio reaches the full FIRE target on its own.

### Backend — Supabase

#### Database — PostgreSQL (SQL)

We use a relational SQL database (PostgreSQL, hosted by Supabase) rather than a document-based NoSQL database like MongoDB. Financial data is inherently structured and relational — transactions always belong to an account, accounts always belong to a user, and every record is anchored to a specific date and time. SQL enforces this structure at the database level, preventing inconsistent or orphaned records. MongoDB's schema-free flexibility is a better fit for unstructured or frequently changing data, which is the opposite of what a finance tracker needs. PostgreSQL also makes it straightforward to query aggregates like total balance or monthly spending with simple, readable SQL.

#### Supabase Features

| Feature | Implementation |
|---|---|
| **Authentication** | Email/password registration and login. User session is managed via Supabase Auth JWT tokens, passed automatically with every database request. |
| **Free tier** | Hosted PostgreSQL with 500MB storage and 50,000 monthly active users. No infrastructure setup required. |
| **Row-Level Security (RLS)** | Database-level policies restricting each user to read and write only their own rows across all tables. |
