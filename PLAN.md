# 🔥 FIRE Finance Tracker — Project Plan

## 1. Data Schema

### `users`
A single table for identity + FIRE profile. Managed by Supabase Auth, extended with onboarding fields.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | Auto-generated by Supabase Auth |
| `name` | `text` | User's display name |
| `email` | `text` | Unique, used for login |
| `age` | `int` | Entered during onboarding |
| `annual_expenses` | `numeric(12,2)` | Estimated annual spending |
| `annual_part_time_income` | `numeric(12,2)` | Expected part-time income in semi-retirement |
| `fire_goal` | `text` | e.g. `"barista"`, `"lean"`, `"fat"`, `"coast"` |
| `created_at` | `timestamptz` | Default: now() |

> **Barista FIRE Number** is a computed field, not stored:
> `(annual_expenses - annual_part_time_income) × 25`

---

### `accounts`
A user may have multiple financial accounts.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `user_id` | `uuid` | FK → `users.id` |
| `type` | `text` | `"checking"` \| `"savings"` \| `"investment"` \| `"other"` |
| `current_balance` | `numeric(12,2)` | Updated whenever a transaction is added |
| `is_active` | `bool` | Default: `true` |
| `created_at` | `timestamptz` | Default: now() |

> **Total Portfolio Balance** = SUM of all active `accounts.current_balance` for a user.

---

### `transactions`
Individual income/expense events.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `account_id` | `uuid` | FK → `accounts.id` |
| `user_id` | `uuid` | FK → `users.id` |
| `amount` | `numeric(12,2)` | Positive = income, Negative = expense |
| `category` | `text` | e.g. `"food"`, `"rent"`, `"salary"`, `"investment"` |
| `description` | `text` | Free-text note |
| `transaction_date` | `date` | When the transaction occurred |
| `created_at` | `timestamptz` | When the record was created |

> When a transaction is inserted, `accounts.current_balance` is updated automatically.

---

### Entity Relationship Diagram

```
users
  └── accounts      (1:N — one user, many accounts)
        └── transactions (1:N — one account, many transactions)
```

### Backend — Supabase

#### Database — PostgreSQL (SQL)

We use a relational SQL database (PostgreSQL, hosted by Supabase) rather than a document-based NoSQL database like MongoDB. Financial data is inherently structured and relational — transactions always belong to an account, accounts always belong to a user, and every record is anchored to a specific date and time. SQL enforces this structure at the database level, preventing inconsistent or orphaned records. MongoDB's schema-free flexibility is a better fit for unstructured or frequently changing data, which is the opposite of what a finance tracker needs. PostgreSQL also makes it straightforward to query aggregates like total balance or monthly spending with simple, readable SQL.

#### Supabase Features

| Feature | Implementation |
|---|---|
| **Authentication** | Email/password registration and login. User session is managed via Supabase Auth JWT tokens, passed automatically with every database request. |
| **Free tier** | Hosted PostgreSQL with 500MB storage and 50,000 monthly active users. No infrastructure setup required. |
| **Row-Level Security (RLS)** | Database-level policies restricting each user to read and write only their own rows across all three tables. |
